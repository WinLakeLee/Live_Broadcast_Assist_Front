# 백엔드 처리 제안 (API Contract 개선안)

프론트엔드 작업 중, 백엔드에서 처리하거나 API 스펙을 변경하면 훨씬 견고하고 유지보수하기 좋을 것으로 예상되는 항목들을 정리한 문서입니다.

## 1. 배송 주소 구조화 (Structured Address)

### 현행 (As-Is)
현재 API(`OrderStatusData`, `CheckoutDraft` 관련 주문 API)는 주소를 하나의 통짜 문자열(`string`)로 주고받습니다.
- 예: `"address": "서울특별시 강남구 테헤란로 152 (강남파이낸스센터) 101동 202호"`

### 문제점
- 프론트엔드에서 수정한 주소를 다시 폼(5개 칸)으로 쪼개서 보여주기 매우 까다롭습니다.
- 향후 통계(지역별 주문량 등)를 내거나, 특정 배송 불가 지역(도서산간 등)을 판별할 때 문자열 파싱의 오류율이 높습니다.

### 개선 제안 (To-Be)
주소를 5개의 상세 필드를 가진 객체(JSON) 형태로 분리하여 주고받는 것을 권장합니다.

```json
"address": {
  "province": "서울특별시",          // 도/광역시/특별시
  "city": "강남구",                 // 시/군/구
  "street": "테헤란로 152",         // 도로명/지번
  "building": "강남파이낸스센터",    // 건물명 (선택)
  "detail": "101동 202호",         // 상세주소
  "zipcode": "06236"              // (선택) 우편번호
}
```
*프론트엔드는 현재 자체적으로 이 5개 필드를 묶어 통짜 문자열로 백엔드에 보내고 있습니다. 백엔드 스펙이 변경되면 즉시 프론트엔드도 구조화된 데이터를 보내도록 수정할 수 있습니다.*

## 2. 주소 유효성 검증 (Address Validation)

### 현행 (As-Is)
프론트엔드가 내장된 행정안전부 행정구역 데이터(`korea-regions.ts`)를 사용하여 1차적으로 사용자가 없는 '도'나 '시/군/구'를 입력하지 못하도록 막고 있습니다.

### 개선 제안 (To-Be)
- **백엔드 검증 추가**: API 단에서도 들어오는 `province`와 `city` 데이터가 실제 존재하는 대한민국의 행정구역인지 확인하는 유효성 검사가 추가되면 보안 및 데이터 무결성에 큰 도움이 됩니다.
- **우편번호/도로명 검증**: (필요 시) 우편번호(Zipcode)를 필수로 받아야 하는 경우, 백엔드에서 외부 배송/택배사 API와 연동하여 해당 주소가 실제 택배 배송이 가능한 도로명 주소인지 검증 후 주문을 확정하는 로직이 이상적입니다.

## 3. 재고 부족 시 정책 (Stock Policy)

### 현행 (As-Is)
사용자가 `partial`(부분 배송 허용) 또는 `all_or_nothing`(전체 구매 취소) 정책을 선택하고 있습니다.

### 개선 제안 (To-Be)
- 재고 부족으로 `all_or_nothing` 정책에 의해 주문이 취소된 경우, 백엔드에서 **"어떤 상품의 재고가 부족해서 전체 취소되었는지"** 그 사유(품절된 상품명 등)를 프론트엔드에 상세한 에러 코드와 함께 응답해 주면, 유저에게 더 친절한 안내 창을 띄워줄 수 있습니다. 
- (현재는 API Error Message를 그대로 노출하거나 일반적인 "요청을 완료하지 못했습니다"로 표시됨)
